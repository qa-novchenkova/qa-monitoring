# Разбор проекта QA Monitoring (конспект для понимания)

> Учебный конспект: что представляет собой проект, как части связаны между собой,
> какими командами всё запускается и что учтено.

---

## 1. О чём проект

`qa-monitoring` — учебный QA-проект, демонстрирующий два навыка:
**автоматизированное тестирование веб-приложения (Selenium)** и **мониторинг доступности
сайтов с визуализацией в Kibana**. Запуск автоматизирован через GitHub Actions (прогон тестов
при изменениях в коде и проверка сайтов по расписанию) и дополнен уведомлениями в Telegram.

---

## 2. Два направления

Проект состоит из двух независимых частей. У каждой своя задача, инструменты и место для отчёта.

| | НАПРАВЛЕНИЕ 1: Автотесты | НАПРАВЛЕНИЕ 2: Мониторинг |
|---|---|---|
| **На какой вопрос отвечает** | «Работает ли логика сайта?» | «Доступен ли сайт и как быстро отвечает?» |
| **Тип проверки** | Глубокие сценарии (логин, корзина) | Внешняя проверка доступности (UP/DOWN) |
| **Чем запускается** | `pytest` | `python monitor/monitor.py` |
| **Главные инструменты** | Selenium, pytest, Page Object | requests, Elasticsearch, Kibana |
| **Где результат (отчёт)** | **Allure** (локально + GitHub Pages) | **Kibana** дашборд (локально) |
| **Объект тестирования** | saucedemo.com (учебный магазин) | qa-dev-toolbox + saucedemo |

Кратко: **тесты → Allure**, **мониторинг → Kibana**. Это разные «табло» под разные задачи.

---

## 3. Карта проекта

### 3.1. Как движутся данные

**Локально (на компьютере):**
```
Автотесты:   pytest (Selenium) --> Allure (отчёт)
Мониторинг:  monitor.py --> лог-файл + Elasticsearch --> Kibana (дашборд)
```

**В облаке (GitHub Actions):**
```
При пуше:      tests.yml   --> прогон автотестов --> Allure-отчёт на GitHub Pages
По расписанию: monitor.yml --> проверка сайтов --> (если падение) Telegram-алерт
```

Облако GitHub не имеет доступа к локальной Kibana (она работает в Docker на компьютере).
Поэтому Kibana наполняется локальными запусками монитора, а в портфолио показывается скриншотами.

### 3.2. Что в чём (дерево файлов)

```
qa-monitoring/
├─ .github/workflows/
│  ├─ tests.yml          # CI: автотесты + публикация Allure на GitHub Pages
│  └─ monitor.yml        # CI: мониторинг сайтов по расписанию (cron) + Telegram
├─ docker/
│  └─ docker-compose.yml # описание контейнеров Elasticsearch + Kibana
├─ pages/                # Page Object — описания страниц сайта
│  ├─ __init__.py        #   делает папку Python-пакетом (пустой)
│  ├─ login_page.py      #   страница входа
│  └─ inventory_page.py  #   страница товаров
├─ tests/                # автотесты
│  ├─ test_login.py      #   вход: успех / неверный пароль
│  └─ test_cart.py       #   добавление товара в корзину
├─ monitor/
│  ├─ monitor.py         # монитор доступности + Telegram-алерт
│  └─ targets.yaml       # конфиг: список сайтов для проверки
├─ conftest.py           # общая фикстура pytest: запуск/закрытие браузера
├─ pytest.ini            # настройки pytest (где искать модули)
├─ requirements.txt      # список Python-библиотек проекта
└─ .gitignore            # что не коммитится (venv, логи, кэш)
```

## 4. Глоссарий (технические определения, аналогия — в скобках)

**QA / тестирование**
- **Автотест** — программа, которая воспроизводит действия пользователя в интерфейсе и проверяет результат без участия человека.
- **Selenium** — библиотека для программного управления реальным браузером: открытие страниц, клики, ввод текста, чтение содержимого.
- **WebDriver** — драйвер-посредник, через который Selenium отдаёт команды конкретному браузеру (Chrome).
- **pytest** — фреймворк для запуска тестов и проверки утверждений; находит тесты, выполняет их и формирует итог.
- **Локатор** — правило поиска элемента на странице (по `id`, CSS-селектору, классу и т.п.).
- **assert** — оператор проверки: если выражение ложно, тест считается упавшим.
- **Page Object** — паттерн проектирования автотестов: каждая страница описывается классом с её локаторами и действиями; локаторы хранятся в одном месте, тесты не дублируют их.
- **Фикстура (fixture)** — функция pytest, готовящая ресурс до теста и освобождающая его после (здесь — запуск и закрытие браузера).
- **Flaky-тест** — тест с нестабильным результатом: падает не из-за дефекта, а из-за внешних факторов (чаще всего тайминга/скорости загрузки).
- **implicit / explicit wait** — механизмы ожидания готовности элемента: общее ожидание для всех поисков (implicit) и точечное ожидание конкретного условия (explicit). Защита от flaky-тестов.
- **headless** — режим браузера без графического окна; применяется на серверах, где нет дисплея.
- **Allure** — инструмент построения отчётов по автотестам: статусы, шаги, вложения, история (тренд) прогонов.

**Мониторинг / данные**
- **Мониторинг доступности** — периодическая внешняя проверка состояния сайта: отвечает ли он и за какое время.
- **HTTP-статус (код ответа)** — числовой код результата запроса (200 — успех, 404 — не найдено, 503 — недоступен и т.д.).
- **UP / DOWN** — производный статус проверки: UP — сайт ответил кодом < 400; DOWN — ошибка, таймаут или нет ответа.
- **Структурированный лог** — запись события в машиночитаемом формате (JSON с полями), пригодная для автоматической обработки, в отличие от свободного текста.
- **Elasticsearch** — поисково-аналитическое хранилище документов (JSON), обеспечивающее быстрый поиск и агрегацию по большим объёмам данных (по аналогии — «склад с быстрым поиском»).
- **Индекс (index)** — именованная коллекция документов в Elasticsearch (по аналогии — «таблица/папка» для однотипных записей); здесь индекс `site-monitoring`.
- **Kibana** — веб-интерфейс для исследования и визуализации данных Elasticsearch: дашборды, графики, фильтры (по аналогии — «витрина» поверх склада).
- **Data View** — настройка в Kibana, указывающая, какие индексы Elasticsearch отображать и какое поле считать временной меткой (по аналогии — «очки», через которые Kibana смотрит на данные).
- **ELK** — связка Elasticsearch + Logstash + Kibana; в проекте используются Elasticsearch и Kibana (без Logstash).

**Инфраструктура / автоматизация**
- **Docker** — платформа для запуска приложений в изолированных контейнерах с уже включёнными зависимостями, без ручной установки в систему.
- **Образ (image)** — неизменяемый шаблон приложения со всеми зависимостями, из которого создаются контейнеры.
- **Контейнер (container)** — изолированный запущенный экземпляр образа со своим процессом и файловой системой.
- **docker-compose** — инструмент описания нескольких контейнеров в одном YAML-файле и управления ими одной командой.
- **Volume** — постоянное хранилище данных контейнера, переживающее его перезапуск/пересоздание (здесь — данные Elasticsearch).
- **CI/CD** — автоматический прогон проверок и действий на сервере при изменениях в коде (непрерывная интеграция/доставка).
- **GitHub Actions** — встроенная в GitHub система CI/CD; сценарии хранятся в `.github/workflows/`.
- **Workflow** — YAML-файл, описывающий, какие шаги и при каких событиях выполняются на сервере GitHub.
- **cron** — синтаксис расписания запусков (например, каждые 30 минут).
- **Artifact (артефакт)** — файлы, сохраняемые прогоном CI для последующего скачивания (здесь — данные Allure, лог монитора).
- **GitHub Pages** — хостинг статических страниц от GitHub (здесь — публикация HTML-отчёта Allure по живой ссылке).
- **venv** — изолированное окружение Python для одного проекта: его библиотеки не пересекаются с системными и с другими проектами.
- **Переменная окружения** — внешний параметр процесса, задаваемый вне кода (используется для секретов и флагов: `HEADLESS`, `TELEGRAM_TOKEN` и т.д.).
- **YAML (.yml)** — текстовый формат конфигураций, где вложенность задаётся отступами.

## 6. Подробный разбор по этапам (как собрать проект заново)

Для каждого этапа: **цель → действия и команды → ключевые файлы → результат.**

### Этап 0. Подготовка окружения
**Цель:** создать структуру проекта, изолированное окружение Python, поднять Elasticsearch + Kibana.
**Действия:**
```bash
mkdir qa-monitoring && cd qa-monitoring     # создать папку проекта и перейти в неё
mkdir docker tests monitor pages            # подпапки под части проекта
python -m venv venv                         # создать виртуальное окружение
source venv/Scripts/activate                # включить его (Git Bash на Windows)
git init                                    # начать отслеживать проект в Git
```
**Зачем что:**
- `mkdir ...` — создаёт папку проекта и подпапки: `docker` (конфиг контейнеров), `tests` (автотесты), `monitor` (монитор), `pages` (Page Object).
- `python -m venv venv` — создаёт **виртуальное окружение** — отдельный контейнер для библиотек только этого проекта. Нужно, чтобы устанавливаемые пакеты (Selenium, requests и т.д.) не попадали в систему и не конфликтовали с другими проектами. Появляется папка `venv/`.
- `source venv/Scripts/activate` — **включает** этот контейнер в текущем терминале: после этого `pip` и `python` работают внутри неё (в начале строки появляется `(venv)`). Без активации команды пойдут в системный Python.
- `git init` — превращает папку в Git-репозиторий, чтобы версионировать код и позже отправить его на GitHub.
- Создать **`.gitignore`** — список того, что **не** хранить в Git (`venv/`, `logs/`, кэш): эти файлы большие и у каждого свои.
- Установить **Docker Desktop**, запустить, дождаться статуса Running — это движок, запускающий контейнеры. Без запущенного Docker Desktop команды `docker` не работают.
- Создать **`docker/docker-compose.yml`** — описание двух контейнеров: `elasticsearch` (хранилище логов, порт 9200; `single-node` — один узел; безопасность отключена для учебной среды; память ограничена 1 ГБ) и `kibana` (витрина с графиками, порт 5601).
```bash
cd docker
docker compose up -d                        # поднять Elasticsearch + Kibana в фоне
```
- `docker compose up -d` — читает `docker-compose.yml`, при первом запуске скачивает образы и запускает контейнеры в фоне (флаг `-d`).
- Проверка: `http://localhost:9200` (Elasticsearch отдаёт JSON о кластере), `http://localhost:5601` (открывается Kibana, прогрев 1–2 минуты).
**Файлы:** `docker/docker-compose.yml`, `.gitignore`.
**Результат:** окружение готово; хранилище (Elasticsearch) и витрина (Kibana) подняты, но пока пусты.

### Этап 1. Первый автотест (понять принцип Selenium)
**Цель:** убедиться, что автотест может управлять браузером.
**Действия:**
- В `requirements.txt` добавить `selenium`, `pytest`; установить:
```bash
pip install -r requirements.txt
```
- Создать `tests/test_login.py` с одним тестом: открыть `saucedemo.com`, ввести логин/пароль, проверить переход на страницу товаров.
```bash
pytest tests/test_login.py -v
```
**Зачем что:**
- `requirements.txt` — текстовый список библиотек проекта; `pip install -r` ставит их все разом в активное venv. Так любой человек одной командой получит те же зависимости (воспроизводимость).
- `selenium` управляет браузером, `pytest` запускает тест и проверяет `assert`.
- `pytest ... -v` — запустить тест в подробном режиме (видно имя теста и статус). Драйвер для Chrome Selenium 4 скачивает сам.
**Файлы:** `requirements.txt`, `tests/test_login.py`.
**Результат:** Chrome открывается сам, выполняет вход, тест проходит (PASSED).

### Этап 2. Page Object + фикстуры + основные тесты
**Цель:** перевести автотесты в профессиональную структуру.
**Действия:**
- Создать `pytest.ini` со строкой `pythonpath = .` (чтобы работал импорт из `pages`).
- Создать `conftest.py` с фикстурой `driver` (запуск браузера до теста, закрытие после).
- Создать Page Object: `pages/__init__.py` (пустой), `pages/login_page.py`, `pages/inventory_page.py` (локаторы и действия страниц).
- Переписать `tests/test_login.py` (успешный вход, вход с неверным паролем) и создать `tests/test_cart.py` (добавление товара в корзину).
```bash
pytest -v                                   # запустить все тесты
```
**Зачем что:**
- `pytest.ini` — файл настроек pytest; `pythonpath = .` добавляет корень проекта в пути поиска, иначе строка `from pages.login_page import LoginPage` не сработает.
- `conftest.py` — особый файл, который pytest подхватывает автоматически; здесь живёт **фикстура** `driver`, открывающая браузер до теста и закрывающая после. Это убирает дублирование кода из каждого теста.
- **Page Object** (`pages/`) — каждый класс описывает одну страницу: её локаторы и действия. Если на сайте поменяется `id` — правка в одном месте, а не во всех тестах.
- `pages/__init__.py` — пустой файл-маркер, чтобы Python считал папку пакетом для импорта.
- `pytest -v` без указания файла прогоняет **все** тесты проекта.
**Файлы:** `pytest.ini`, `conftest.py`, `pages/*`, `tests/*`.
**Результат:** локаторы вынесены в Page Object, браузер управляется фикстурой, проходят 3 теста.

### Этап 3. Отчёты Allure
**Цель:** наглядная отчётность по автотестам.
**Действия:**
- В `requirements.txt` добавить `allure-pytest`; установить.
- Установить Allure CLI (требует Java):
```bash
npm install -g allure-commandline
allure --version
```
- Прогнать тесты с записью данных и открыть отчёт:
```bash
pytest --alluredir=allure-results
allure serve allure-results
```
**Зачем что:**
- Allure состоит из двух частей: **`allure-pytest`** (Python-библиотека, пишет «сырые» данные прогона в папку) и **Allure CLI** (утилита на Java, строит из этих данных HTML-отчёт).
- `pytest --alluredir=allure-results` — прогнать тесты и сложить данные в папку `allure-results`.
- `allure serve allure-results` — собрать из данных красивый отчёт и открыть его в браузере.
**Файлы:** `requirements.txt` (обновлён).
**Результат:** визуальный отчёт со статусами, графиками, историей.

### Этап 4. Монитор доступности
**Цель:** универсальная проверка доступности сайтов.
**Действия:**
- В `requirements.txt` добавить `requests`, `PyYAML`; установить.
- Создать `monitor/targets.yaml` — список сайтов (имя + URL).
- Создать `monitor/monitor.py` — читает конфиг, делает HTTP-запрос к каждому сайту, замеряет код и время ответа, формирует статус UP/DOWN, пишет структурированный JSON в консоль и в `logs/monitor.log`.
- Добавить `logs/` в `.gitignore`.
```bash
python monitor/monitor.py
```
**Зачем что:**
- `requests` — библиотека HTTP-запросов; `PyYAML` — чтение конфига `.yaml`.
- `targets.yaml` — список сайтов **вынесен из кода в конфиг**: добавить сайт = дописать `name + url`, код не трогается. Это и делает инструмент универсальным (подключается к любому проекту).
- `monitor.py` пишет **структурированный JSON** (а не просто текст), потому что дальше эти данные пойдут в Elasticsearch, и машине удобно читать именно такой формат.
- `logs/` в `.gitignore` — логи не коммитятся (они генерируются и у каждого свои).
**Файлы:** `monitor/targets.yaml`, `monitor/monitor.py`.
**Результат:** в консоли и в `logs/monitor.log` появляются JSON-записи проверок. Новый сайт добавляется правкой только `targets.yaml` — код не меняется (инструмент универсальный).

### Этап 5. Логи → Elasticsearch → дашборд Kibana
**Цель:** хранить результаты проверок и строить по ним графики.
**Действия:**
- В `requirements.txt` добавить `elasticsearch` (версия под 8.x); установить.
- Дополнить `monitor.py` отправкой каждой проверки в индекс `site-monitoring` Elasticsearch.
- Несколько раз запустить монитор, чтобы накопить данные:
```bash
for i in {1..10}; do python monitor/monitor.py; sleep 30; done
```
- Проверить, что данные есть: `http://localhost:9200/site-monitoring/_count`.
- В Kibana создать **Data View** `site-monitoring` (timestamp = `timestamp`), затем в **Dashboard** построить графики: время ответа по сайтам и количество UP/DOWN.
**Зачем что:**
- `elasticsearch` — Python-клиент, через который монитор отправляет данные в Elasticsearch (в **индекс** `site-monitoring` — «папку» для однотипных записей).
- Цикл `for ... sleep 30` — запустить монитор 10 раз с паузой 30 сек, чтобы на графике во времени было что показать (одна точка — неинформативна).
- `_count` — служебный запрос Elasticsearch: показывает число записей в индексе (проверка, что данные долетели).
- **Data View** — «очки» Kibana: говорят ей, какой индекс показывать и какое поле считать временем. Без него графики не построить.
**Файлы:** `monitor/monitor.py` (обновлён), `requirements.txt`.
**Результат:** дашборд в Kibana с метриками доступности и времени ответа.

### Этап 6. Автотесты в CI (GitHub Actions)
**Цель:** автоматический прогон тестов на сервере при каждом пуше.
**Действия:**
- В `conftest.py` добавить headless-режим (по переменной `HEADLESS`), т.к. на сервере нет экрана.
- Создать `.github/workflows/tests.yml` (установка Python, зависимостей, прогон `pytest`).
- Создать на GitHub репозиторий и отправить код:
```bash
git add -A && git commit -m "..." && git push
```
**Зачем что:**
- **headless** — режим браузера без окна; нужен потому, что на сервере GitHub нет экрана. Включается переменной окружения `HEADLESS`, которую workflow задаёт только в CI (локально окно остаётся видимым).
- **workflow** `tests.yml` лежит в `.github/workflows/` — GitHub ищет сценарии строго там. Он описывает шаги: скачать код, поставить Python и зависимости, прогнать `pytest`.
- `git add -A && git commit -m "..." && git push` — сохранить все изменения и отправить на GitHub; пуш запускает CI.
**Файлы:** `conftest.py` (обновлён), `.github/workflows/tests.yml`.
**Результат:** во вкладке Actions при каждом пуше прогоняются автотесты.

### Этап 7. Публикация Allure-отчёта на GitHub Pages
**Цель:** живая онлайн-ссылка на отчёт.
**Действия:**
- В `tests.yml` добавить установку Allure CLI, генерацию отчёта (`allure generate`) и публикацию официальными actions (`upload-pages-artifact` + `deploy-pages`).
- В Settings → Pages выбрать источник **GitHub Actions**.
```bash
git add -A && git commit -m "..." && git push
```
**Зачем что:**
- В CI нужно сгенерировать готовый HTML-отчёт (`allure generate`), а не только «сырые» данные — поэтому в workflow ставится Allure CLI.
- `upload-pages-artifact` + `deploy-pages` — официальные шаги GitHub, которые публикуют папку с отчётом на **GitHub Pages** (хостинг статических страниц). Этот способ надёжнее ручной публикации в ветку.
- Источник Pages = **GitHub Actions** — обязательная настройка для этого способа публикации.
**Файлы:** `.github/workflows/tests.yml` (обновлён).
**Результат:** отчёт доступен по адресу `https://qa-novchenkova.github.io/qa-monitoring/`.

### Этап 8. Мониторинг по расписанию (cron)
**Цель:** автоматическая проверка сайтов без участия человека.
**Действия:**
- Дополнить `monitor.py`: флаг `ENABLE_ES` (в облаке отправка в локальный ES отключается) и выход с кодом 1 при наличии DOWN (красный прогон = сигнал).
- Создать `.github/workflows/monitor.yml` с расписанием `cron: "*/30 * * * *"`.
```bash
git add -A && git commit -m "..." && git push
```
**Зачем что:**
- `ENABLE_ES` — переменная-флаг: в облаке локальный Elasticsearch недоступен, поэтому отправку туда отключаем (не засоряя лог ошибками).
- Выход с кодом 1 при DOWN — стандартный приём: ненулевой код = «ошибка», и прогон в Actions становится красным. Это и есть сигнал-алерт «сайт упал».
- **cron** `"*/30 * * * *"` — расписание «каждые 30 минут» (поля: минута час день месяц день_недели, время в UTC).
**Файлы:** `monitor/monitor.py` (обновлён), `.github/workflows/monitor.yml`.
**Результат:** GitHub сам проверяет сайты каждые 30 минут; при падении прогон краснеет.

### Этап 9. Уведомления в Telegram
**Цель:** оповещение о падении сайта.
**Действия:**
- Создать бота через `@BotFather`, получить токен; узнать `chat_id`.
- Добавить в `monitor.py` функцию `send_telegram` и отправку при наличии DOWN.
- Токен и `chat_id` задавать через переменные окружения (локально) и **GitHub Secrets** (в CI); в `monitor.yml` передать их в блоке `env`.
**Зачем что:**
- **Токен** — «ключ» бота, **chat_id** — адрес чата, куда слать сообщение. Сообщение отправляется обычным HTTP-запросом (`requests`) к Telegram Bot API.
- Токен и chat_id хранятся в **секретах**, а не в коде: локально — в переменных окружения, в CI — в **GitHub Secrets** (`${{ secrets.* }}`). Это правило безопасности — секреты нельзя вписывать в репозиторий.
**Файлы:** `monitor/monitor.py` (обновлён), `.github/workflows/monitor.yml` (обновлён).
**Результат:** при падении сайта приходит сообщение в Telegram (надёжно — из GitHub Actions).

## 8. Основные команды (шпаргалка)

```bash
# --- Виртуальное окружение ---
source venv/Scripts/activate          # включить venv (Git Bash на Windows)

pytest -v                             # запустить все тесты
pytest --alluredir=allure-results     # прогон + сбор данных для Allure

# --- Docker (Elasticsearch + Kibana) ---
cd docker
docker compose up -d                  # поднять контейнеры в фоне

docker ps                             # посмотреть запущенные контейнеры

# --- Мониторинг ---
python monitor/monitor.py             # проверить сайты из targets.yaml

# --- Git: сохранить и отправить на GitHub ---
git add -A && git commit -m "описание" && git push   # "запушить изменения" (запускает CI)
```

Локальные адреса: Elasticsearch — `http://localhost:9200`, Kibana — `http://localhost:5601`.

## 10. Что учтено в плане безопасности

- **Токен бота и chat_id — в секретах**, не в коде (локально через переменные окружения,
  в CI через **GitHub Secrets** `${{ secrets.* }}`).
- Секреты GitHub **зашифрованы**, не отображаются в интерфейсе, **маскируются в логах** и
  **не передаются форкам** — безопасны даже в публичном репозитории.
- При утечке токена — **немедленный revoke** в `@BotFather` и выпуск нового.
- **`.gitignore`** исключает из репозитория лишнее (`venv`, логи, кэш).
- Монитор **устойчив к сбоям**: недоступность Elasticsearch или Telegram не роняет программу
  (обёрнуто в `try/except`).
- В учебной среде безопасность Elasticsearch отключена осознанно; для публичного развёртывания
  её необходимо включить (логины/пароли, TLS).

## 11. Запуск проекта

### 11.1. С нуля (на новой машине)
```bash
git clone <URL репозитория>
cd qa-monitoring
python -m venv venv
source venv/Scripts/activate          # Windows-Bash; Linux/Mac: source venv/bin/activate
pip install -r requirements.txt

# Автотесты (нужен установленный Chrome):
pytest -v

# Мониторинг + Kibana (нужен Docker Desktop):
cd docker
docker compose up -d 
python monitor/monitor.py
# открыть http://localhost:5601 и создать Data View "site-monitoring"
```

### 11.2. Ежедневный сценарий

**Что нужно проверить ЛОКАЛЬНО (автотесты + Kibana):**
1. Запустить **Docker Desktop** и дождаться статуса Running (после перезагрузки он часто не стартует сам, и контейнеры остановлены — это нормально).
2. Поднять Elasticsearch + Kibana заново (данные не пропали — они в volume):
```bash
cd docker
docker compose up -d
docker ps                             # убедиться, что qa-elasticsearch и qa-kibana запущены
cd ..
```
3. Открыть `http://localhost:5601` (Kibana прогревается 1–2 минуты).
4. Включить окружение и работать:
```bash
source venv/Scripts/activate
python monitor/monitor.py             # добавить свежие данные в Kibana
pytest -v                             # прогнать автотесты
```

> Чтобы контейнеры поднимались автоматически вместе с Docker Desktop, можно один раз добавить
> в `docker-compose.yml` каждому сервису строку `restart: unless-stopped` — тогда повторный
> `docker compose up -d` после старта Docker будет не нужен.


Итог: «локальное» (Kibana, локальные прогоны) требует запущенного Docker и venv; «облачное»
(CI, Pages, Telegram) живёт на серверах GitHub и доступно всегда.
